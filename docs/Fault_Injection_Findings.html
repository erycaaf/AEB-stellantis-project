TL;DR

A injeção de falhas sistemática como parte da validação cruzada ASIL-D identificou seis condições em que o módulo PID viola requisitos de robustez estabelecidos pela ISO 26262. O módulo Alert passou nos 18 de 18 testes de robustez.

Enquadramento

Este achado é parte esperada e positiva do processo de validação independente definido pela ISO 26262-6 §5.4.8. O propósito do cross-assignment é justamente encontrar defeitos que a equipe de implementação original não detectou com seus testes nominais. Sugere-se tratar os 6 achados como tarefas de correção a serem absorvidas pela Deliverable 4/4.

02 Finding #1 — Propagação de NaN

FI-PID-001 NaN propaga para a saída do PID
Severidade: ALTA aeb_pid.c:138

Descrição

Quando qualquer entrada do controlador é NaN (Not-a-Number), o valor propaga pela cadeia aritmética completa e atinge a variável de saída output->brake_pct como NaN. Em IEEE 754, qualquer operação aritmética envolvendo NaN produz NaN, e a função clamp_f32 existente não detecta essa condição porque NaN falha em todas as comparações (<, >, ==).

Evidência reproduzível

$ make fault

[FAULT-A1] a_ego = NaN (faulty accelerometer)
  FAIL: brake_pct stays in [0,100] (no NaN propagation)
  FAIL: brake_bar stays in [0,10] (no NaN propagation)

[FAULT-A4] decel_target = NaN (corrupted FSM output)
  FAIL: brake_pct stays in [0,100] with decel_target=NaN

Cenário realista

Um acelerômetro defeituoso pode emitir NaN quando sua rotina de calibração executa divisão por zero interna. Se esse valor chegar ao PID, a saída do controlador fica indefinida. O atuador de frenagem, ao receber NaN, pode: (a) interpretar como zero e não frear; (b) interpretar como valor máximo e travar rodas; (c) entrar em estado de erro e desligar. Nenhum desses comportamentos é previsível — violando diretamente o princípio de determinismo da ISO 26262 ASIL-D.

Causa raiz

/* aeb_pid.c — linha 138 */
error = decel_target - a_ego;
                       ↑
                       // Se a_ego for NaN, error = NaN
                       // Todas as operações subsequentes = NaN

Patch proposto

#include <math.h>  /* para isfinite() */

/* No início de pid_brake_step, após o NULL check (linha 102): */

/* Defensive finite-value check — ASIL-D robustness */
if (!isfinite(decel_target) || !isfinite(a_ego))
{
    /* Inputs inválidas: fail-safe (sem frenagem) + reset do estado */
    s_integral       = 0.0F;
    s_prev_brake_pct = 0.0F;
    output->brake_pct = 0.0F;
    output->brake_bar = 0.0F;
}
else
{
    /* ... resto do código existente ... */
}
Teste de aceitação

Após aplicar a correção, os testes FAULT-A1, FAULT-A2, FAULT-A3 e FAULT-A4 devem passar, e os testes nominais existentes devem continuar passando (sem regressão).

03 Finding #2 — FSM state corrompido

FI-PID-002 fsm_state inválido causa frenagem espúria
Severidade: CRÍTICA aeb_pid.c:119

Descrição

A guarda que decide se o controlador deve atuar é:

if ((fsm_state < FSM_BRAKE_L1) || (decel_target <= 0.0F))
{
    /* reset e saída zero */
}
else
{
    /* calcula frenagem */
}

O problema: a verificação fsm_state < FSM_BRAKE_L1 (onde FSM_BRAKE_L1 = 3) aceita qualquer valor maior ou igual a 3. Valores como 7, 42, 99, 255 são interpretados como se fossem estados de frenagem válidos, fazendo o PID calcular e produzir comando de frenagem.

Evidência reproduzível

[FAULT-C1] fsm_state = 99 (invalid state)
  FAIL: invalid fsm_state produces zero braking (fail-safe)
  FAIL: invalid fsm_state produces zero pressure

[FAULT-C2] fsm_state = 255 (UINT8_MAX, bit-flip scenario)
  FAIL: fsm_state=255 produces zero braking (fail-safe)

Cenário realista

Um único bit-flip em RAM decorrente de radiação cósmica, EMI, ou glitch de tensão poderia transformar o estado FSM_STANDBY (1) em FSM_STANDBY | 0x80 (129). Hoje, esse valor faria o PID computar frenagem com decel_target indefinido — potencialmente uma frenagem de emergência em linha reta em estrada livre. ISO 26262 ASIL-D exige proteção contra exatamente este tipo de falha single-event upset.

Patch proposto

/* aeb_pid.c — substituir linha 119 */

/* ANTES (inseguro): */
if ((fsm_state < (uint8_t)FSM_BRAKE_L1) || (decel_target <= 0.0F))

/* DEPOIS (seguro): */
const uint8_t is_braking_state =
    ((fsm_state == (uint8_t)FSM_BRAKE_L1) ||
     (fsm_state == (uint8_t)FSM_BRAKE_L2) ||
     (fsm_state == (uint8_t)FSM_BRAKE_L3) ||
     (fsm_state == (uint8_t)FSM_POST_BRAKE));

if ((!is_braking_state) || (decel_target <= 0.0F))
{
    /* reset e saída zero */
}

A lógica muda de blacklist implícita (< X) para whitelist explícita (conjunto de valores válidos). Qualquer bit-flip agora cai automaticamente no ramo fail-safe, exatamente como ASIL-D exige.

Teste de aceitação

Após aplicar, FAULT-C1 e FAULT-C2 devem passar (brake_pct = 0). Os testes nominais (test_pid.c) devem continuar passando — a whitelist cobre exatamente os estados que já eram testados.

04 O que foi testado e passou

Para o registro, reforço que 12 dos 18 testes do PID e todos os 18 do Alert passaram. Em particular, os seguintes cenários foram considerados robustos:

Cenário testado Resultado
a_ego = +Infinity (sensor saturado) PASS Saída em [0, 100]%
a_ego = -Infinity (sensor travado baixo) PASS Saída em [0, 100]%
a_ego = FLT_MAX (limite numérico) PASS Saída em [0, 100]%
decel_target = 1000 m/s² (demanda absurda) PASS Saturação em 100%
Jerk limiter mesmo com comando em degrau PASS |Δ| ≤ max_delta
Reset do integrator ao passar por estado inválido PASS Sem carry-over
alert_map com qualquer estado fora da faixa 0-6 PASS Default silent
override_detect com todas variações patológicas PASS 11/11

Isto demonstra que partes significativas do controle já estão corretamente protegidas — em especial o jerk limiter, a saturação por clamp_f32 para valores finitos, e toda a lógica do Alert.
